# PROJECT_IDEA.md — Motivational Bot: Полная концепция

---

## Проблема

У друзей наступил этап **«эмоционального выгорания и потери вектора»**. Это не просто лень — это системный отказ психики двигаться к результату. На старте был дофамин и розовые очки, но как только начались серые будни — дисциплина посыпалась. Нет внешней точки опоры, которая бы возвращала в строй каждый день.

---

## Решение

Бот становится **персональным цифровым надзирателем и ментором**, который не просто спамит текстом, а создаёт у человека структуру дня, из которой невозможно выпасть.

Система перекрывает три фазы активности:

### Фаза 1: «Боевой подъём» (Утро)
- Сообщение приходит в точное время подъёма
- Содержит **«инъекцию реальности»**: зачем он проснулся и что потеряет, если продолжит лежать
- Цель: переключить мозг из режима комфорта в режим задачи
- Тон: жёсткий, конкретный, без сюсюканья

### Фаза 2: «Фокус на цель» (День)
- Сообщения в моменты дневного спада мотивации
- Не ванильные цитаты — **удары по конкретным болям и амбициям** (деньги, статус, бренд, успех)
- Роль: голос совести, перехватывающий внимание в момент слабости
- Не даёт «слиться» под предлогом усталости

### Фаза 3: «Дофаминовая подпитка» (Вечер)
- Аудит результатов дня
- За каждое выполненное действие — **порция одобрения и признания**
- Закрепляет нейронную связь: «Сделал дело — получил признание — я крутой»
- Если результата нет — лёгкий «осадок» недовольства собой, чтобы завтра встать злее

## Целевая аудитория

Личный круг друзей автора. Каждый — с конкретными амбициями (бизнес, деньги, статус, личный бренд), у каждого — свой контекст и болевые точки.

---

## План запуска: Три этапа

### Этап 1: «Железный ритм» (MVP)
**Цель**: запустить автономную систему доставки смыслов.

- Бот внедряется в ЛС каждого друга
- Три волны сообщений в день: утро / день / вечер
- Жёсткое расписание: конкретное время, конкретный человек, конкретный текст
- Тестирование контента: какие слова заставляют встать, какие пролистывают
- Стек: Python 3.11, APScheduler + PostgreSQL jobstore, python-telegram-bot v20, Supabase

**Результат**: у каждого друга есть ощущение, что в его жизни появился «голос дисциплины», который не затыкается.

---

### Этап 2: «Центр управления» (Админ-панель)
**Цель**: перейти от автоматики к гибкому ручному управлению.

Функционал:
- Изменение времени отправки на лету (без перезапуска бота)
- Настройка частоты и интенсивности сообщений под «температуру» мотивации группы
- Включение режимов: «похвала» (все летят) / «штурм» (все затухли)
- Просмотр списка друзей и их индивидуальных настроек
- Добавление/редактирование текстов сообщений через интерфейс

Реализация:
- Telegram inline-кнопки (admin-only команды)
- Роль Admin_ID — только ты имеешь доступ к панели

**Результат**: ты становишься кукловодом, который точечно подстраивает систему под реальный контекст жизни.

---

### Этап 3: «Цифровой разум» (Нейросеть-консалтинг)
**Цель**: превратить бота из «кричалки» в инструмент решения проблем.

Механика:
- Кнопка **«Нужна помощь»** — экстренный вызов
- Включается Claude API (Anthropic), обученный на бизнес-кейсах и контексте
- Бот анализирует затык, раскидывает его по полочкам, выдаёт алгоритм действий
- Переход: мотиватор → бизнес-партнёр

Сценарии использования:
- Не знает, где найти поставщика
- Не понимает, как настроить рекламу
- Загнался и не видит выхода
- Нужен чёткий план на день/неделю

**Результат**: бот не просто поднимает, но и помогает выйти из конкретного ступора делом, а не словом.

## Риски и решения

| Риск | Решение |
|------|---------|
| Бот не отправляет сообщения вовремя | APScheduler с `misfire_grace_time`, systemd auto-restart, логи каждой отправки |
| Размытое расписание, сообщения теряются | В коде/БД: конкретная дата + время + получатель + текст. Никаких «примерно» |
| Контент перестаёт цеплять | Ротация пулов сообщений, A/B по тону, замена неэффективных текстов через админку |
| Бот блокируется другом | Обработка ошибки `bot was blocked`, флаг `is_active=False` в БД |
| Railway перезапускает контейнер | APScheduler хранит джобы в Supabase — при рестарте всё восстанавливается автоматически |
| Deploy сбивает расписание | `misfire_grace_time=120`: APScheduler догоняет пропущенные задачи после рестарта |

```
- 08:00  →  Утреннее сообщение #1 (мягкий старт → жёсткий финал)

```

## Принципы написания сообщений

1. **Конкретика, не абстракция** — не «иди к мечте», а «через 3 месяца без действий — ты на том же месте»
2. **Боль перед выгодой** — сначала то, что теряет, потом то, что получит
3. **Личное обращение** — по имени, под контекст человека
4. **Короткие удары** — не лекция, а одна острая мысль
5. **Чередование тонов** — жёсткий / поддерживающий / провокационный

## Метрики успеха (неформальные)

- Друзья перестали игнорировать сообщения
- Хотя бы один сообщил о реальном действии после утреннего сообщения
- Ни одного пропуска отправки по расписанию
- Ты, как автор, чувствуешь контроль над системой через панель

## Техническая спецификация

### Инфраструктура
```
GitHub (main branch)
    ↓  push → auto-deploy
Railway (Python worker, polling)
    ↓  DATABASE_URL
Supabase (PostgreSQL)
    ↑  jobstore
APScheduler (джобы хранятся в БД, выживают рестарты)
```

### Почему такой стек
- **Railway вместо VPS**: нет DevOps-боли, авто-деплой из GitHub, Railway сам рестартует при падении
- **Supabase с первого дня**: Railway не имеет persistent disk — SQLite невозможен. Supabase даёт PostgreSQL бесплатно на старте
- **APScheduler + SQLAlchemy jobstore**: ключевое архитектурное решение. Все джобы хранятся в таблице `apscheduler_jobs` в Supabase. После каждого деплоя/рестарта Railway — расписание живёт. Без этого при каждом деплое все задачи слетали бы
- **Polling, не Webhook**: для MVP проще; бот в первую очередь отправляет сам, не ждёт входящих

### Поток данных (Happy Path)
```
1. Admin добавляет друга через /adduser → запись в таблицу users
2. Admin задаёт расписание → запись в schedule_slots
3. При старте бота (main.py) → читаем все активные schedule_slots → создаём APScheduler jobs
4. В назначенное время APScheduler вызывает send_message(user_id, phase)
5. Из таблицы messages берём случайное сообщение нужного phase
6. Отправляем через Telegram API
7. Пишем результат в delivery_log (sent / failed / blocked)
```

### CI/CD Pipeline
```
Локальная разработка → git push origin main
                              ↓
                        GitHub repo
                              ↓ webhook
                        Railway build (Nixpacks)
                              ↓
                        python main.py
```
Никаких GitHub Actions на старте. Railway сам слушает GitHub.

### Локальный запуск (dev)
```bash
git clone <repo>
cd <repo>
cp .env.example .env        # заполнить BOT_TOKEN, DATABASE_URL, ADMIN_TELEGRAM_ID
pip install -r requirements.txt
python main.py
```
